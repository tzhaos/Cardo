# Worklog: Native Messaging frame size cap (30788051)

Date: 2026-07-13  
Branch: `feature/hover-tip`  
Addresses review F1 / F10: NM stdin length-prefixed frames had no body max (local OOM DoS).

## Goal

Cap Native Messaging stdin frame body size (same order as HTTP JSON body: 2 MiB), reject oversize from the length prefix alone without allocating a huge body buffer, surface a typed error, reset the stdin buffer, and log the oversize diagnostic at most once per process.

## Changes

### `src/native-host/messageCodec.ts`

- Exported `MAX_NATIVE_MESSAGE_BYTES = 2 * 1024 * 1024` (local constant; not shared with HTTP).
- Exported `NativeMessageTooLargeError` with `code: 'native_message_too_large'` and `bodyLength`.
- `tryDecodeNativeMessage`: after reading the 4-byte LE length prefix, if `bodyLength > MAX_NATIVE_MESSAGE_BYTES`, throw immediately (no wait for full frame, no body allocation).
- Valid frames at exactly max body size still decode as before.

### `src/native-host/main.ts`

- Catch `NativeMessageTooLargeError` separately from generic invalid JSON.
- Write `{ ok: false, errorMessage }` response.
- Reset `pending` buffer.
- Diagnostic `message-too-large bodyLength=… code=…` logged once per host process (`loggedMessageTooLarge` flag) to avoid flood if stdin keeps advertising oversize.

### `src/native-host/messageCodec.test.ts`

- Oversize: header-only with `MAX + 1` and `0xffff_ffff` both throw `NativeMessageTooLargeError` (proves no body needed).
- At-max: complete frame with body length exactly `MAX_NATIVE_MESSAGE_BYTES` still decodes.
- Existing round-trip and partial-frame tests retained.

## Out of scope

- Desktop / web-next / runtime HTTP body cap (already exists).
- Sharing a constant with `httpServer.ts` (kept local per task).
- `openLocalResource` path normalization (separate finding).
- encode-side max (host-controlled responses).
