# Frontend review (30788051)

Date: 2026-07-13  
Scope (read-only): `src/web/**`, official theme packs / recipes, HoverTip, AGENTS frontend norms, `messages.ts`, motion ownership, `themeId` branches, hardcoded hex, error-screen, Desktop tray copy.  
Norms: `Agents.md` frontend section + `docs/architecture/theme-pack-authoring.md`.  
Product code was not modified.

---

## Executive judgment

The web-next shell is directionally compliant with Cardo architecture: single `--cardo-*` token surface, five registered official themes, Settings window opacity-only entry, box/resize drag commits Command only on pointer end, and main chrome copy is bilingual via `messages.ts`.

It is not fully gate-clean against AGENTS frontend norms. Residual debt concentrates in five places:

1. Shared product CSS still hardcodes accent / danger / status hex that ignore live theme tokens.
2. Component TS still branches on `themeId === 'fluent' | …` for geometry and chrome dialect that belongs in tokens or recipe CSS.
3. HoverTip is correctly implemented and used on bottom/history toolbars, but many box/top-bar chrome controls still use native `title=` as the primary tip path.
4. Fatal error-screen and Desktop tray violate copy rules (schema `9` literal, monorepo runbook, Chinese-only tray, Architecture jargon).
5. Several text-heavy panels still enter with Motion `scale`, which reopens the Fluent-style blur lesson for list/confirm UI.

Verdict: mergeable only with known debt; P0 copy + token hex should land before calling the frontend norms “done.” Theme registry itself is healthy; adaptation risk is mostly bypass hardcode and TS dialect, not missing packs.

## Findings

### F1 — HoverTip implementation is correct; adoption is incomplete

Implementation (`src/web/ui/cardo/hover-tip.tsx` + primitives tooltip + design-system CSS):

- Product API wraps Radix Tooltip with `asChild` trigger (ref-safe for MotionButton).
- App root provides `TooltipProvider` in `WebNextApp.tsx` (default delay 320ms / skip 120ms).
- Visual class `.cardo-hover-tip` uses `--cardo-create-background` / `--cardo-create-text` (theme-aware dark speech bubble).
- Enter animation is opacity-only (`cardo-ui-tooltip-in`); comment and CSS match text-shell rule.
- Arrow fill matches tip background; `pointer-events: none` avoids sticky hover fights.
- `IconButton.tooltip` correctly composes HoverTip.

Gaps:

| Gap | Evidence |
| --- | --- |
| Native `title` still primary on box chrome | `BaseBoxFrame.tsx` promote / appearance / lock / view toggles use `title={t(...)}` with `aria-label`, not HoverTip |
| Collection + canvas tools same pattern | `CollectionPage.tsx`, `CanvasToolsToolbar.tsx` use `title=` |
| Top bar tools use `title` on some controls | `TopBar.tsx` canvas tool buttons ~375 |
| Bottom / history toolbars are the good path | Only 6 `tooltip=` call sites: BottomToolbar (4) + HistoryToolbar (2) |
| HoverTip does not default a “short” delay | Comment prefers short delay; unset → Provider 320ms. Acceptable but slightly slow for chrome |

Correctness conclusion: HoverTip itself is product-ready. Norm failure is dual tip strategy (native title vs bubble) on the same product surface family.

### F2 — `themeId` branches in business/chrome components

Allowed under AGENTS only as rare shell capability with comments. Current branches:

| Location | Branch | Assessment |
| --- | --- | --- |
| `BaseBoxFrame.tsx` | BOX_CORNER_RADIUS map for all 5 ids | Geometry dialect reimplemented in TS; should be token or CSS var (`--cardo-box-radius`) |
| `BottomToolbar.tsx` | fluent labeled create + chip 36 vs 40 + search width 280/360 | Shell dialect without enough “why” comments; chip size belongs in recipe/token |
| `TopBar.tsx` | `isFluent = themeId === 'fluent'` | Used for chrome layout dialect |
| `SettingsPanel.tsx` | `isSwiftUI` traffic lights; static nav for glass/fluent/material/swiftui | Traffic lights: acceptable platform chrome with recipe CSS; static nav has good blur-avoidance comment |
| `SettingsNavIcons.tsx` | Fluent color icons vs ThemeIcon | Acceptable theme-glyph module pattern |

Risk: every new official theme requires TS edits in boxes and toolbars, defeating “recipe + token only” expansion.

High impact (accent / danger / status not re-skinned by packs or user overrides):

| Path | Issue |
| --- | --- |
| `styles/add-views.css` | `.cardo-add-primary { background: #60a5fa }` — not `--cardo-blue` |
| `styles/boxes.css` | Delete confirm `#ff3045` vs item delete `var(--cardo-red)` — dual danger reds |
| `styles/shell.css` | Runtime banner reconnect `#d97706`, disconnect `#dc2626` |

Legitimate hex remains in pack JSON, `defaultTokens`, color presets, and self-contained error-screen styles.

### F4 — Official theme registration is sound

- SoT: `OFFICIAL_BUILT_IN_THEME_IDS` = classic, glass, fluent, material, swiftui.
- Recipes mapped in `OFFICIAL_THEME_RECIPE_ENTRIES`.
- Builtin JSON under `themes/builtin/<id>/theme.cardo-theme.json`.
- Recipe CSS under `src/web/styles/themes/…`.
- `builtInPacks.ts` fails hard if pack id ∉ official set or missing.

No missing official id. Adaptation risk is content of recipes/shared CSS, not registry gaps.

### F5 — error-screen copy and version drift

`src/web/ui/cardo/error-screen.ts`:

- Has en/zh and surface-aware steps (good structure).
- Hardcodes `schemaVersion is 9` instead of `DATABASE_SCHEMA_VERSION` (Desktop main already interpolates the constant).
- Default recovery steps are monorepo developer runbooks (`npm run desktop:build`, `artifacts/cli/cardo.js`, mixing checkouts). AGENTS: product recovery for end users; technical detail secondary/collapsed.
- Titles/summaries correctly say “Cardo Runtime” in several paths — borderline product jargon; prefer “本机服务 / local service” for user-facing titles where possible, keep Runtime in technical details.
- Self-contained hex palette is intentional isolation when app CSS failed; acceptable second visual system for fatal UI only.

### F6 — Desktop tray copy gaps (outside web-next but in review scope)

`src/desktop/main.ts` `updateTrayMenu`:

- Labels Chinese-only: 「隐藏/显示 Cardo」「退出」「退出并停止 Runtime」.
- Exposes “Runtime” in user menu (architecture term).
- Tooltip is English brand “Cardo” only — OK.
- No locale coupling to preferences.

Violates AGENTS: tray must not be Chinese-only hardcode; formal product wording.

### F7 — i18n surface quality

Strengths:

- `WEB_NEXT_MESSAGES` en/zh key parity complete.
- Main interactive UI uses `useI18n().t` / `translateWebNext`.
- Runtime banner messages go through i18n keys.

Weaknesses:

- User-visible jargon still in messages used by Settings About / Features:
  - `settings.tokenThemePack`: “Design tokens + Theme Pack”
  - `settings.updateDescription` / `settings.updateIdle`: “GitHub milestone”, “Desktop build”
  - `settings.featuresDescription`: “shell tools”
  - `settings.feature.chrome.runtimeBannerDescription`: “Runtime connection toast”
- Dead keys: `layout.exitZen`, unused history op labels, unfinished settings keys (CSS snippet / density / etc.) — see worklogs hardcode-audit-i18n.
- `settings.tokenThemePack` is rendered in About (`SettingsPanel.tsx`).

### F8 — Drag / Command mid-frame

Verified pass:

- Box resize: motion values on move; `updateBoxFrame` → `fireCommand` only in `onEnd` (`BaseBoxFrame.tsx`).
- Box drag: `updateBoxDragFrame` mutates uiStore without Command; commits on release via `BoxPageDropController` / `updateBoxFrame` / `moveBoxToPage`.
- `previewBoxOnPage` is optimistic projection only (no `dispatchDatabaseCommand`).
- Settings window drag/resize: motion values + `Math.round` on end.

This is the correct ownership model and should be preserved under any geometry refactors.

## Theme adaptation risks

1. Glass settings shell hardcode  
   Recipe paints opaque-ish custom hex + backdrop blur instead of pure `var(--cardo-settings-chrome)`. User settingsChrome overrides and token validation intent are weakened. Historical “settings底无效 / 糊” class of bug can return on Glass only.

2. Shared CTA / delete / banner hex  
   Classic-looking blue and dual reds mean Fluent/Material/SwiftUI accent packs never fully recolor primary actions or danger confirms. Visual “theme works on chrome, not on dialogs” regression risk.

3. TS geometry maps vs pack radius tokens  
   Box compact/idle radius and toolbar chip sizes diverge from CSS vars. Motion interpolates numeric radius from `BOX_CORNER_RADIUS` (comment notes CSS vars unreliable for Motion) — valid constraint, but values should still be derived from a single SoT (e.g. pack radius numbers mirrored into a small table generated from tokens, not free-hand per id).

4. Fluent-specific chrome in TS  
   Labeled “New” create button and search open width depend on `isFluent`. A future pack that wants the same dock language must edit React. Prefer data attributes from pack options / chrome profile.

5. Settings nav Fluent color icons  
   Acceptable; ensure Material/SwiftUI/Glass stay on ThemeIcon path (currently true). Avoid stacking `filter: drop-shadow` on these 20px glyphs (currently not observed on nav).

6. layoutId indicators  
   Classic settings still uses `layoutId="settings-nav-indicator"`; themed shells correctly use static CSS. Tab indicator freezes layout spring during box drag — good mitigation. Do not reintroduce layoutId under Fluent settings.

## Priority fixes

P0 (user-visible / version drift / theme-break high)

1. error-screen: interpolate `DATABASE_SCHEMA_VERSION`; split packaged-user recovery vs developer detail; drop monorepo-first steps as default.
2. Desktop tray: locale-aware formal labels; remove bare “Runtime” from primary menu wording (e.g. “退出并停止本机服务”).
3. Tokenize shared CSS: add primary (`--cardo-blue` / create tokens), delete confirm (`--cardo-red`), runtime banner status tokens; unify dual danger reds.
4. Glass settings shell: `background` from `--cardo-settings-chrome` (keep atmospheric gradients as overlays if needed).

P1 (norm compliance / maintainability)

5. Migrate box / collection / canvas chrome from `title=` to `IconButton.tooltip` or `HoverTip` (keep `aria-label`).
6. Collapse `themeId` geometry (box radius, chip size) into token-backed numbers or recipe CSS + data attributes; leave only documented shell capability branches.
7. Rewrite live jargon strings: shell tools, Design Token + Theme Pack, GitHub milestone, Runtime toast → product language.
8. Prune dead keys (`layout.exitZen`, unused history labels, orphan settings keys) or wire real UI.

P2 (polish)

9. Remove residual scale on GlobalSearchPanel / ItemDeleteView / CollectionPage entry (opacity + optional y only).
10. Align HoverTip default delay with product “short chrome tip” intent if 320ms feels laggy.
11. Settings selection rings: replace classic blue rgba with `color-mix(... var(--cardo-blue) ...)`.
12. Item type accent map: optional semantic tokens if packs should recolor type chips.

Verification when fixing (local, in order):

```text
npm run format
npm run format:check
npm run lint
npm run lint:eslint
npm run validate:themes
npm run build:all
```

Visual gate after theme CSS changes: classic / glass / fluent / material / swiftui × light/dark, at least settings, top bar, box, bottom toolbar, dropdown; rebuild surface and open the new artifact path (do not judge against stale Programs\Cardo installs).
